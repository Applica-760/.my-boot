- 前提のアイデア
    - promot->context->herness->loop エンジニアリングという変遷が指摘されている
    - Loopエンジニアリングにおいて、ループの基準、受け入れ条件などをどう定義するかが質を大きく左右
    - 自然言語での指示やSkillsでは負債が蓄積する可能性
    - 静的解析ツールベースのループ定義が有用になるはず（CIのようにYamlでルールを設定できるとか）
        - LLMの精度は非常に高く意図を汲み取る能力は向上し続けている
        - しかし原理的に確率的な振る舞いを排除できない
        - だからこそ決定論的なルールがLoppのためには重要
    - 不安：「AIが賢くなって自然言語で曖昧な指示も通るようになった」という価値と衝突する考えではないか？

- 目的
    - LLMへの自然言語指示だけに依存せず、アーキテクチャ規約を静的に検査できる仕組みを作る
    - AIが生成・変更したコードに対して、依存方向・import制約・循環依存・責務境界違反を機械的に検出する
    - Skillsは「手順・作法」を担い、MCPサーバは「厳格な判定」を担う
    - **AI coding loopの中で、実装 → 静的検査 → 違反修正 → 再検査を自動化できるようにする**

- 基本方針
    - LLMに「守れているか」を判断させない
    - LLMには「検査結果を読んで修正する」役割を持たせる
    - アーキテクチャ規約はYAML/JSONなどの機械可読な形式で定義する
    - 検査結果はJSON/SARIFなど、CI・IDE・LLMが共通利用できる形式で返す

- 想定ユースケース
    - AIが実装したコードが、定義済みアーキテクチャ規約に違反していないか検査する
    - 依存グラフを生成し、循環依存や不正な依存方向を可視化する
    - domain層がinfra層に依存していないか検査する
    - 特定モジュールが禁止されたライブラリをimportしていないか検査する
    - 特定の責務が特定の境界内に閉じているかを検査する
    - AIに違反内容を返し、修正ループを回す

- 実装方針
    - MCPが有効なのではないか？
        - AIプロバイダ非依存、可搬
        - Skillsは自然言語指示で非決定論的
    - ゼロから作らず、既存のLinter, Code Graph, CI系のツールを統合 
        - 使えるものは再開発しない
        - でも、「これ入れれば全部入る」ようになればまとめる価値は高い
    - MCPサーバの役割
        - 静的解析ツール群を統合する
        - 言語ごとの依存グラフを構築する
        - YAML/JSONで定義されたアーキテクチャルールを評価する
        - 違反箇所・違反理由・影響範囲・修正ヒントを返す
        - 依存グラフや違反グラフを出力する
        - CI、IDE、LLMクライアントから共通して呼び出せる検査APIを提供する
