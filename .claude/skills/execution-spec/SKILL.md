# 実行仕様書Skill

## 概要
別の実装者やエージェントに渡しても意図が伝わる実行仕様書を作成する。
実装の詳細手順を過剰に縛らず、実装者がコードベースに合わせて判断できる余地を残す。

## 基本ルール
- 実装はしない
- 仕様書は単体で読める独立した文書として書く
- 推測で要件を補完しない。不明点は明示する
- 対象範囲と非対象範囲を必ず分ける
- 受け入れ基準は検証可能な形で書く
- 1仕様書 = 1実装単位（1PR/1エージェントセッションで完結する粒度）
- 複数の独立した変更を含む場合は仕様書を分割する
- 実行方法は扱わない。具体的な実行は利用者側の運用に委ねる
- 実行はサンドボックス環境で行い、読み書き権限はタスクの性質に応じて最小限に指定する

## 規約
- `template.md` を参照し、下記の出力構成で作成する
- 各項目は原則1行。箇条書きで簡潔に記述する
- 「だ、である」体で統一する
- mdのリスト`-`、チェックボックス`- [ ]`、コードブロック`` ` ``を適切に用いる
- ファイルパス・既存仕様への参照は根拠として明記する
- 特定ツール、実行環境、エージェント固有の内部指示を含めない

## 出力先
- ファイルパス規約: `.claude/specs/YYYYMMDD_HHmm_<topic>.md`
- 上記に作成する

## 出力構成
- **目的**
- **背景・前提**
- **対象範囲**
- **非対象範囲**
- **変更要件**
- **実装方針** — 構成・設計判断の方向性まで。メソッド名・引数等のコードレベル詳細は書かない
- **受け入れ基準**
- **検証条件**
- **リスク・注意点**
- **不明点・確認事項**

## 手順
1. ユーザーの依頼から、目的・成果物・対象範囲・制約・検証方法・不明点を抽出する
2. 必要に応じてコードベース、既存ドキュメント、関連設定を確認する
3. 確認した事実と推測を混ぜず、根拠がある内容だけを仕様へ反映する
4. 次をチャットに出力し、ユーザーの確認を得る
   - 仕様書ドラフト
   - 情報不足・不明点
   - ユーザー判断が必要な点（粒度分割、方針の選択肢等）
5. ユーザーの回答を反映し、仕様書を確定する
